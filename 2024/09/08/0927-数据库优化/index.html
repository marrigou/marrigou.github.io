【学习】SQL优化及查慢查询

优化表结构和索引

索引是提高查询性能的最有效手段之一。通过为查询条件中的列创建索引，可以大幅减少扫描的数据量，提升查询效率。

  • 创建索引：
    对频繁作为查询条件的字段创建索引，尤其是 WHERE 条件、JOIN 子句、排序（ORDER BY）、分组（GROUP BY）字段。
  • 组合索引：
    如果查询条件中涉及多个列，可以创建组合索引，提高查询效率。

避免不必要的排序和分组

  • 优化排序和分组：排序和分组操作通常会消耗大量资源，特别是在大表中。
    • 使用索引可以加速 ORDER BY 和 GROUP BY 操作。
    • 尽量减少使用 ORDER BY RAND() 这样的操作，因为会导致全表排序。

减少子查询，使用连接

  • 优化子查询：子查询可能会导致多次扫描数据，尽量将子查询改写为 JOIN，这样可以避免重复的扫描。
    示例：将子查询改为 JOIN：

分解复杂查询

  • 查询分解：将一个复杂的大查询拆分为多个小查询，减少单次查询的负担，同时可以更高效地利用索引。
    例如，将复杂的多表关联查询拆解为多个单表查询，并将中间结果缓存。

使用缓存

  • 缓存机制：对于一些经常被查询的数据，可以使用缓存（如 Redis、Memcached）来减少对数据库的直接访问，提升性能。

分页查询

  • 分页查询：对于返回大量数据的查询，应尽量使用分页（LIMIT）控制每次返回的结果数量，避免一次性返回大量数据。

避免使用 SELECT *

  • 指定具体字段：避免使用 SELECT *，而是明确指定需要查询的字段。这样可以减少不必要的数据传输，提升查询效率。

不同的数据库（如 MySQL、PostgreSQL、SQL Server）中 EXPLAIN 的输出格式可能略有不同，但主要信息大同小异。以下以 MySQL 为例，介绍常见的 EXPLAIN 输出信息。

2. EXPLAIN 输出字段详解

执行 EXPLAIN 后，输出结果通常包含以下字段：

1. id

id 是查询中每个子查询或查询块的标识符，值越大优先级越高。通常，id 为 NULL 的表示是最外层查询，数值越高的 id 代表嵌套的子查询或者复杂查询中的每个步骤。

2. select_type

表示查询的类型，常见的类型包括：

  • SIMPLE：表示简单查询，不涉及子查询和联合查询。
  • PRIMARY：最外层的查询。
  • SUBQUERY：子查询。
  • UNION：UNION 中的第二个及后续查询。
  • DEPENDENT SUBQUERY：依赖外部查询的子查询。

3. table

查询中涉及的表名或子查询的别名。如果使用了连接操作，会依次显示涉及到的每个表。

4. type

这是最重要的字段之一，表示连接类型，直接影响到查询性能。常见的值及含义：

  • system：表只有一行（等同于系统表），这是最理想的状态。
  • const：表示通过索引查找，可以快速返回唯一的一行数据，通常发生在 PRIMARY KEY 或 UNIQUE 索引查询时。
  • eq_ref：对于每一行，从另一表中最多返回一行数据。这通常是对主键或唯一索引的等值查询。
  • ref：非唯一索引扫描，返回匹配某个值的所有行。
  • range：通过索引进行范围扫描，通常出现在 BETWEEN、> 、< 操作时。
  • index：全索引扫描（full index scan），意味着查询需要扫描整个索引来找出符合条件的行，但不需要访问表。
  • ALL：全表扫描，这是最糟糕的查询情况，性能最低。
    优化提示：尽量避免 ALL 类型的全表扫描，可以通过创建索引或优化查询条件来提升性能。

5. possible_keys

可能用到的索引。这个字段显示了查询优化器认为可能有助于优化查询的索引。如果 possible_keys 是 NULL，表示没有可用的索引，可能需要创建新的索引来提升查询效率。

6. key

实际使用的索引。如果该字段为 NULL，表示没有使用任何索引。理想情况下，应该有一个索引被选中。如果没有使用索引，说明查询效率较低，可以考虑优化查询或者添加索引。

7. key_len

使用的索引部分的长度（字节数）。这个字段可以帮助我们判断索引的选择是否合理。理论上，key_len 越小越好，因为这意味着索引覆盖的数据量较小，查询代价更低。

8. ref

表示查询使用索引时，哪个列或常量与索引匹配。通常用于 eq_ref 和 ref 类型的查询中，指示如何使用索引来查找匹配的行。

9. rows

优化器估计需要扫描的行数，通常是基于表的统计信息进行估算。这一值可以帮助我们大致判断查询的开销。如果 rows 数量非常大，可能意味着查询效率低，可能需要优化。

10. filtered

表示条件过滤掉的数据百分比。100 表示所有数据都满足条件，10 表示只有10%的数据满足查询条件。该值较低时，可能意味着查询条件不够精确。

11. Extra

该字段包含额外的执行信息。常见的值包括：

  • Using index：表示查询只使用了索引进行查找，数据不需要访问表，即“覆盖索引查询”，这是非常高效的。
  • Using where：表明需要通过 WHERE 条件进行过滤。此时可能没有使用到索引，或者索引未能完全覆盖查询。
  • Using temporary：表示查询需要创建临时表，通常出现在使用 GROUP BY 或 ORDER BY 时。临时表会导致查询性能下降。
  • Using filesort：表示 MySQL 需要对结果进行额外的排序操作，而不是使用索引顺序。可以通过调整索引或修改查询语句避免文件排序（filesort）。

3. 查看 EXPLAIN 输出后的优化步骤

通过 EXPLAIN 查看查询的执行计划后，我们可以识别性能瓶颈并做出调整。

1. 避免全表扫描（type=ALL）

如果 type 字段为 ALL，表示发生了全表扫描，这通常是性能问题的主要原因之一。
优化策略：

  • 添加索引：检查 possible_keys 和 key 字段，是否有索引可以加速查询。通常可以对 WHERE 子句中的列添加索引，特别是频繁查询或参与过滤条件的列。
  • 使用更高效的查询条件：尽量使用可以利用索引的查询条件，比如避免使用 LIKE '%abc%' 这种不利用索引的模糊查询。

2. 优化连接（type=ref 或 eq_ref）

当多个表连接时，type 字段为 eq_ref 或 ref 是较好的连接方式，这表明查询使用了索引和主键。
优化策略：

  • 确保关联列有索引：在 JOIN 查询中，确保被关联的列上有索引。 可以在 department_id 或 id 上添加索引。

3. 减少 Using temporary 和 Using filesort

如果 Extra 字段出现 Using temporary 或 Using filesort，说明数据库在查询过程中创建了临时表或执行了额外的排序操作，这会降低查询性能。
优化策略：

  • 优化排序：如果 ORDER BY 子句触发了 Using filesort，可以尝试将排序字段放入索引中。
  • 减少临时表的使用：使用合理的索引来避免创建临时表，尤其是 GROUP BY 和 DISTINCT 语句，确保相关列已经被索引覆盖。

4. 使用覆盖索引（Using index）

当 Extra 字段显示 Using index 时，表示查询只通过索引就能完成，而无需访问表的数据。这个状态是非常高效的，称为“覆盖索引”查询。
优化策略：

  • 创建合适的组合索引：确保查询使用的所有列都被索引覆盖，这样数据库就不需要访问实际表的数据。

5. 避免不必要的列选择

SELECT * 会选择表的所有列，这会增加查询的负担，尤其是大表。如果只需要少量列，应该明确指定这些列。
优化策略：

  • 只选择必要的列：减少查询中返回的列数量，特别是在联合查询和子查询中。

怎么查慢 SQL

慢 SQL 是指那些执行时间较长、影响系统性能的 SQL 语句。为了保证数据库的效率，识别并优化慢 SQL 是非常重要的。

查找慢 SQL

MySQL 提供了多种方式来查找慢 SQL，最常用的是通过慢查询日志（Slow Query Log）来记录执行时间较长的 SQL 语句。

启用慢查询日志

在 MySQL 中，默认情况下，慢查询日志是关闭的。你可以通过以下步骤启用慢查询日志：

  • 修改 MySQL 配置文件：
    在 MySQL 的配置文件（如 my.cnf 或 my.ini）中，添加以下配置来启用慢查询日志：

    • slow_query_log: 设置为 1 以启用慢查询日志。
    • slow_query_log_file: 设置慢查询日志的保存路径。
    • long_query_time: 定义慢查询的执行时间阈值，单位为秒，超过该时间的查询会被记录。
    • log_queries_not_using_indexes: 设置为 1，会记录未使用索引的查询，即使它们执行时间较短。

SHOW PROCESSLIST 命令可以显示当前 MySQL 中正在执行的 SQL 语句，包括执行时间较长的 SQL。该命令适合实时查看哪些 SQL 在消耗资源。

  • 输出显示所有正在运行的查询，可以通过 Time 列判断哪些 SQL 执行时间过长。

使用 EXPLAIN 分析 SQL 执行计划

一旦发现慢 SQL，可以使用 EXPLAIN 关键字来查看 MySQL 是如何执行该查询的。EXPLAIN 会显示查询的执行计划，包括是否使用了索引、表的访问方式等。

执行计划中的常见字段解释：

  • type：访问类型，值越好性能越高，常见类型从好到差依次为：const、eq_ref、ref、range、index、ALL（ALL 表示全表扫描，性能最差）。
  • possible_keys：查询可能使用的索引。
  • key：实际使用的索引。
  • rows：扫描的行数，越少越好。
